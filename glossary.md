## A

**Agent (Zabbix)**
A lightweight daemon installed on a monitored host that collects local metrics and sends them to the Zabbix server.

**Alertmanager**
A component of the Prometheus ecosystem that handles alerts generated by Prometheus, routing them to email, Slack, PagerDuty, or other notification channels.

**Ansible**
An open-source configuration management and automation tool that uses YAML-based playbooks and connects to hosts over SSH — no agent required.

**Ansible Role**
A reusable, self-contained unit of Ansible automation that organizes tasks, variables, files, and templates into a standard directory structure.

**API (Application Programming Interface)**
A defined interface that allows software systems to communicate with each other — foundational to microservices, cloud services, and CI/CD tooling.

**ArgoCD**
A declarative GitOps continuous delivery tool for Kubernetes that automatically synchronizes cluster state with a Git repository.

**Artifact**
A versioned, deployable output produced by a CI/CD pipeline — e.g., a compiled binary, Docker image, or JAR file.

**Audit Log**
A chronological record of system events and user actions, used for security investigations and compliance.

**Authentication**
The process of verifying the identity of a user or system (e.g., username/password, SSH key, token).

**Authorization**
The process of determining what an authenticated user or system is permitted to do (e.g., RBAC policies).

**Auto-Scaling**
The ability of a system to automatically add or remove compute resources in response to load changes.

## B

**Bash**
The Bourne Again SHell — the default command-line interpreter on most Linux distributions, also widely used for scripting automation.

**Binary**
A compiled, executable file that can be run directly by an operating system.

**Blue-Green Deployment**
A deployment strategy that maintains two identical environments (blue = current, green = new). Traffic is switched to green after validation, enabling instant rollback.

**Branch (Git)**
An independent line of development in a Git repository. Branches allow parallel work without affecting the main codebase.

**Build**
The process of compiling source code and its dependencies into a deployable artifact.

**Bundle**
A packaged collection of files — e.g., a Helm chart bundle or a software release bundle containing binaries and configs.

## C

**Canary Deployment**
A release strategy where a new version is rolled out to a small percentage of users first. If stable, the rollout continues to 100%.

**CD (Continuous Delivery/Deployment)**
- **Continuous Delivery**: Every code change is automatically built and tested, ready to be deployed manually.
- **Continuous Deployment**: Every code change that passes tests is deployed to production automatically.

**CI (Continuous Integration)**
The practice of frequently merging code changes into a shared repository, with automated builds and tests running on every change.

**chmod**
A Linux command to change file and directory permissions (e.g., `chmod 755 script.sh`).

**chown**
A Linux command to change the owner and/or group of a file or directory (e.g., `chown user:group file`).

**Cloud Provider**
A company that offers on-demand computing infrastructure over the internet — the three major providers are AWS, Azure, and GCP.

**Cluster**
A group of servers (nodes) that work together as a single system — used in Kubernetes, databases, and distributed systems.

**ConfigMap (Kubernetes)**
A Kubernetes object used to store non-sensitive configuration data as key-value pairs, injected into pods as environment variables or files.

**Container**
A lightweight, portable unit of software that packages an application and its dependencies together, isolated from the host OS using kernel features (namespaces, cgroups).

**Container Registry**
A storage system for container images (e.g., Docker Hub, GitHub Container Registry, AWS ECR, Quay.io).

**Container Runtime**
The software responsible for running containers (e.g., containerd, CRI-O, runc).

**cron / crontab**
A time-based job scheduler in Linux. `crontab` is the file that defines scheduled commands using a five-field time expression.

**curl**
A command-line tool for transferring data with URLs — commonly used to test HTTP APIs and download files.

**CVE (Common Vulnerabilities and Exposures)**
A public database of known security vulnerabilities in software, each assigned a unique identifier (e.g., CVE-2023-1234).

## D

**Daemon**
A background process on Linux/Unix that runs continuously and performs system tasks (e.g., `sshd`, `dockerd`).

**DAST (Dynamic Application Security Testing)**
Security testing performed against a running application to find vulnerabilities like injection, authentication flaws, and misconfigurations.

**Data Source (Grafana)**
A configured connection in Grafana to a metrics backend — e.g., Prometheus, Zabbix, Elasticsearch, InfluxDB.

**Declarative Configuration**
A style of defining system state by specifying *what* you want (not *how* to achieve it) — used in Kubernetes YAML, Terraform HCL, and Ansible playbooks.

**Deployment (Kubernetes)**
A Kubernetes object that manages a set of identical pod replicas, supporting rolling updates and rollbacks.

**DevOps**
A set of practices, tools, and cultural philosophies that combines software development (Dev) and IT operations (Ops) to shorten the development lifecycle and deliver high-quality software continuously.

**DevSecOps**
An extension of DevOps that integrates security practices into every phase of the development and operations lifecycle.

**DNS (Domain Name System)**
A hierarchical naming system that translates human-readable domain names (e.g., `example.com`) into IP addresses.

**Docker**
A platform for developing, shipping, and running applications in containers using a client-server architecture with a daemon (`dockerd`).

**Docker Compose**
A tool for defining and running multi-container Docker applications using a `docker-compose.yml` file.

**Dockerfile**
A text file containing instructions for building a Docker image, executed layer by layer.

**DRY (Don't Repeat Yourself)**
A software principle encouraging code reuse and avoiding duplication — applied in Ansible roles, Terraform modules, and pipeline templates.

## E

**ELK Stack**
A popular log management stack consisting of **E**lasticsearch (storage/search), **L**ogstash (ingestion/processing), and **K**ibana (visualization).

**Environment Variable**
A dynamic named value stored in a process's environment, used to configure applications without hardcoding values (e.g., `DATABASE_URL`, `API_KEY`).

**Etcd**
A distributed key-value store used by Kubernetes to store all cluster state and configuration data.

**Exporter (Prometheus)**
A small program that collects metrics from a target system and exposes them in Prometheus format (e.g., `node_exporter` for Linux host metrics).

## F

**Failover**
The automatic switching to a standby system, server, or network when the primary one fails.

**Firewall**
A network security system that monitors and controls incoming and outgoing network traffic based on defined rules.

**Fork**
1. In Git: a personal copy of another person's repository.
2. In open source: a project derived from an existing project's source code (e.g., OpenTofu forked from Terraform).

**FQDN (Fully Qualified Domain Name)**
A complete domain name that specifies a host's exact location in DNS (e.g., `web01.prod.example.com`).

## G

**Git**
A distributed version control system for tracking changes in source code and coordinating work among developers.

**GitHub Actions**
A CI/CD platform built into GitHub that uses YAML-based workflow files stored in `.github/workflows/`.

**GitLab CI/CD**
A CI/CD platform built into GitLab, configured via a `.gitlab-ci.yml` file at the repository root.

**GitOps**
An operational framework that uses Git as the single source of truth for declarative infrastructure and application configuration.

**Grafana**
An open-source analytics and visualization platform used to create dashboards from metrics stored in Prometheus, Zabbix, and other data sources.

**grep**
A Linux command-line tool for searching text using patterns (regular expressions) — e.g., `grep "error" app.log`.

## H

**HA (High Availability)**
A system design approach that minimizes downtime by eliminating single points of failure and enabling automatic failover.

**HCL (HashiCorp Configuration Language)**
A declarative configuration language used by Terraform and OpenTofu to define infrastructure resources.

**Helm**
A package manager for Kubernetes that bundles related manifests into reusable, versioned "charts."

**Host (Zabbix)**
A monitored entity in Zabbix — could be a physical server, VM, network device, or application.

**HTTP/HTTPS**
- **HTTP**: Hypertext Transfer Protocol — the foundation of web communication.
- **HTTPS**: HTTP Secure — HTTP encrypted with TLS/SSL.

## I

**IaC (Infrastructure as Code)**
The practice of managing and provisioning infrastructure through machine-readable configuration files rather than manual processes.

**IAM (Identity and Access Management)**
A framework for managing user identities and their permissions to access resources — used in AWS, Azure, GCP, and Kubernetes.

**Idempotent**
A property of an operation that produces the same result whether run once or multiple times — essential for Ansible playbooks and Terraform plans.

**Image (Container)**
A read-only template used to create containers — built from a Dockerfile (Docker) or Containerfile (Podman).

**Ingress (Kubernetes)**
A Kubernetes object that manages external HTTP/HTTPS access to services within a cluster, typically through an Ingress Controller (e.g., Nginx).

**Inventory (Ansible)**
A file or dynamic source that lists the hosts and groups Ansible will manage. Can be in INI or YAML format.

**IP Address**
A numerical label assigned to a device on a network. IPv4 uses 32-bit addresses (e.g., `192.168.1.1`), IPv6 uses 128-bit addresses.

## J

**Jenkins**
An open-source automation server used for building, testing, and deploying software through configurable pipelines defined in `Jenkinsfile`.

**Jenkinsfile**
A text file that contains the definition of a Jenkins pipeline, written in Groovy DSL or declarative syntax.

**Jinja2**
A templating language used in Ansible to dynamically generate configuration files and values using variables and conditionals.

**JSON (JavaScript Object Notation)**
A lightweight data-interchange format widely used in APIs, configuration files, and cloud provider outputs.

## K

**Kibana**
The visualization layer of the ELK Stack — provides dashboards, search, and analytics over data stored in Elasticsearch.

**kubectl**
The command-line tool for interacting with Kubernetes clusters (e.g., `kubectl get pods`, `kubectl apply -f`).

**Kubernetes (K8s)**
An open-source container orchestration system for automating deployment, scaling, and management of containerized applications.

## L

**Latency**
The delay between a request being sent and a response being received — a key performance metric in networking and applications.

**LDAP (Lightweight Directory Access Protocol)**
A protocol for accessing and maintaining distributed directory information — often used for centralized authentication.

**Load Balancer**
A device or software that distributes incoming network traffic across multiple servers to improve availability and performance.

**Logstash**
A server-side data processing pipeline that ingests, transforms, and sends log data to Elasticsearch.

**Loki**
A horizontally scalable log aggregation system by Grafana Labs, designed to work natively with Prometheus and Grafana.

## M

**Manifest (Kubernetes)**
A YAML or JSON file that describes a Kubernetes object's desired state (e.g., a Deployment, Service, or ConfigMap).

**Merge (Git)**
The process of integrating changes from one branch into another.

**Metric**
A quantitative measurement collected over time — e.g., CPU usage percentage, request rate, error count. Used by Prometheus, Zabbix, and Grafana.

**Microservices**
An architectural style where an application is built as a collection of small, independently deployable services.

**Module (Terraform/OpenTofu)**
A reusable, self-contained collection of Terraform/OpenTofu configuration files that define a set of infrastructure resources.

**Mount**
The process of making a filesystem or directory accessible at a specified location in the directory tree. Used in containers for persistent storage (volumes).

## N

**Namespace (Kubernetes)**
A logical partition within a Kubernetes cluster that isolates resources between teams, environments, or applications.

**Namespace (Linux)**
A kernel feature that isolates system resources (processes, network, filesystem) — the foundation of container isolation.

**Node (Kubernetes)**
A physical or virtual machine that runs containerized workloads as part of a Kubernetes cluster.

**Node Exporter**
A Prometheus exporter that exposes Linux/Unix host metrics (CPU, memory, disk, network) for scraping.

## O

**Observability**
The ability to understand the internal state of a system by examining its outputs — the three pillars are metrics, logs, and traces.

**OPA (Open Policy Agent)**
A general-purpose policy engine that enables policy-as-code across the stack — used in Kubernetes via Gatekeeper.

**OpenTofu**
An open-source fork of Terraform, maintained by the Linux Foundation, created after HashiCorp changed Terraform's license to BSL in 2023. Fully compatible with Terraform up to version 1.5.

**Orchestration**
The automated coordination and management of complex systems and services — e.g., Kubernetes orchestrates containers.

## P

**Package Manager**
A tool for installing, updating, and managing software packages — e.g., `apt` (Debian/Ubuntu), `yum`/`dnf` (RHEL/Rocky), `brew` (macOS).

**Pipeline**
An automated sequence of steps (build → test → deploy) that code changes pass through from commit to production.

**Playbook (Ansible)**
A YAML file containing one or more plays that define automation tasks to run on target hosts.

**Pod (Kubernetes)**
The smallest deployable unit in Kubernetes — a group of one or more containers that share network and storage resources.

**Podman**
A daemonless, rootless container engine compatible with Docker CLI commands. Uses OCI-standard images and integrates natively with systemd.

**Policy as Code**
The practice of defining security and compliance rules in code that can be version-controlled and automatically enforced — e.g., OPA, Sentinel.

**Port**
A numerical identifier (0–65535) that distinguishes different services running on the same host — e.g., port 22 (SSH), 80 (HTTP), 443 (HTTPS).

**Prometheus**
An open-source systems monitoring and alerting toolkit that collects metrics by scraping HTTP endpoints at defined intervals.

**PromQL**
Prometheus Query Language — a functional query language used to select, aggregate, and compute Prometheus metrics.

**Provider (Terraform/OpenTofu)**
A plugin that enables Terraform/OpenTofu to interact with a specific API or cloud service (e.g., AWS, Azure, Kubernetes).

**Pull Request (PR)**
A mechanism in Git hosting platforms (GitHub, GitLab) to propose, review, and discuss code changes before merging into the main branch.

## R

**RBAC (Role-Based Access Control)**
A method of controlling access to resources based on the roles assigned to users — used in Kubernetes, cloud IAM, and Linux.

**Replication**
The process of copying data or services across multiple systems for redundancy, availability, or performance.

**Repository (Repo)**
A storage location for code, configuration, and version history — managed by Git and hosted on platforms like GitHub or GitLab.

**Rolling Update**
A deployment strategy that gradually replaces old instances with new ones, with zero downtime.

**RPO (Recovery Point Objective)**
The maximum acceptable amount of data loss measured in time — i.e., how old can a backup be and still be acceptable?

**RTO (Recovery Time Objective)**
The maximum acceptable length of time a system can be offline after a failure before recovery must be complete.

**Runbook**
A documented set of procedures for operating, troubleshooting, and recovering a system — essential for incident response.

## S

**SAST (Static Application Security Testing)**
Security analysis of source code without executing it — identifies vulnerabilities early in the development process.

**SCP (Secure Copy Protocol)**
A command-line tool for securely transferring files between hosts using SSH.

**Secret (Kubernetes)**
A Kubernetes object used to store sensitive information such as passwords, tokens, and SSH keys — base64-encoded (not encrypted by default).

**Secrets Management**
The practice of securely storing, accessing, and auditing sensitive credentials — tools include HashiCorp Vault, AWS Secrets Manager, and Azure Key Vault.

**Service (Kubernetes)**
A Kubernetes object that exposes a set of pods as a stable network endpoint with a consistent IP address and DNS name.

**Service Mesh**
An infrastructure layer that manages service-to-service communication within a microservices architecture — e.g., Istio, Linkerd.

**Shell**
A command-line interpreter that provides a user interface to the operating system (e.g., Bash, Zsh, Fish).

**SNMP (Simple Network Management Protocol)**
A protocol for monitoring and managing network devices — supported by Zabbix for monitoring routers, switches, and printers.

**SSH (Secure Shell)**
A cryptographic network protocol for securely operating network services and remote login over an unsecured network.

**SSH Key**
A cryptographic key pair (public + private) used for passwordless SSH authentication.

**SSL/TLS**
- **SSL**: Secure Sockets Layer (legacy).
- **TLS**: Transport Layer Security — the modern encryption protocol used for HTTPS and other secure communications.

**State (Terraform/OpenTofu)**
A file (`terraform.tfstate`) that records the current state of managed infrastructure, used to determine what changes need to be applied.

**StatefulSet (Kubernetes)**
A Kubernetes object for managing stateful applications (e.g., databases) that require stable network identities and persistent storage.

**Subnet**
A logical subdivision of an IP network — used to organize and isolate resources in cloud VPCs and on-premises networks.

## T

**TCP/IP**
The foundational protocol suite of the internet. **TCP** ensures reliable, ordered delivery; **IP** handles addressing and routing.

**Template (Ansible / Jinja2)**
A file with placeholders that are replaced with actual values at runtime using the Jinja2 templating engine.

**Terraform**
An open-source IaC tool by HashiCorp for provisioning and managing cloud and on-premises infrastructure using HCL. Changed to BSL license in v1.6.

**Trigger (Zabbix)**
A logical expression in Zabbix that evaluates collected metric data and fires an alert when defined thresholds are exceeded.

## U

**UDP (User Datagram Protocol)**
A connectionless transport protocol that prioritizes speed over reliability — used for DNS queries, video streaming, and monitoring agents.

**Uptime**
The percentage of time a system is operational and available. Expressed as SLA percentages (e.g., 99.9% = ~8.7 hours downtime/year).

## V

**Vault (HashiCorp)**
A secrets management tool that provides secure storage, access control, and auditing for sensitive credentials and tokens.

**VPC (Virtual Private Cloud)**
An isolated virtual network within a cloud provider — used to segment and secure cloud resources (AWS VPC, Azure VNet, GCP VPC).

**Volume (Container)**
A persistent storage mechanism that exists outside a container's filesystem, allowing data to survive container restarts.

**VM (Virtual Machine)**
An emulation of a physical computer running an operating system and applications on virtualized hardware.

## W

**Webhook**
An HTTP callback triggered by an event in one system that sends data to another system in real time — used extensively in CI/CD pipelines.

**Worker Node (Kubernetes)**
A node in a Kubernetes cluster that runs containerized workloads (pods), managed by the control plane.

## Y

**YAML (YAML Ain't Markup Language)**
A human-readable data serialization format used extensively in DevOps — Kubernetes manifests, Ansible playbooks, CI/CD pipeline configs, and Docker Compose files are all written in YAML.

## Z

**Zabbix**
An enterprise-grade open-source monitoring platform supporting agent-based monitoring, SNMP, IPMI, and JMX — used for monitoring servers, network devices, applications, and cloud services.

**Zabbix Agent**
A daemon installed on monitored hosts that actively collects local metrics and sends them to the Zabbix server.

**Zabbix Server**
The central component of Zabbix that collects, processes, and stores monitoring data received from agents and other sources.

**Zero Downtime Deployment**
A deployment approach that ensures users experience no service interruption during an application update — achieved via rolling updates, blue-green, or canary strategies.
